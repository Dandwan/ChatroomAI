# 摘要模板

代码摘要存放于 `docs/development-status/summaries/` 下。目录树完全镜像项目源码树结构，每个源文件对应一个 `<文件名>.md` 摘要。

## 目录与命名

- **根目录**：`docs/development-status/summaries/`
- **结构**：完全镜像项目目录树。如果项目有 `src/utils/helper.py`，其摘要位于 `docs/development-status/summaries/src/utils/helper.py.md`。
- **命名**：`<原文件名>.md` — 例如 `main.py` → `main.py.md`，`Dockerfile` → `Dockerfile.md`
- **目录创建**：根据需要创建 `summaries/` 下的中间目录以匹配源码树。

### 排除规则

不要为匹配以下任何模式的文件创建摘要：

| 类别 | 模式 |
|------|------|
| VCS | `.git/` |
| 依赖 | `node_modules/`、`venv/`、`.venv/`、`vendor/`、`__pycache__/` |
| 构建产物 | `dist/`、`build/`、`.next/`、`.nuxt/`、`target/`、`out/` |
| 锁文件 | `*.lock`、`package-lock.json`、`yarn.lock`、`Gemfile.lock`、`Cargo.lock` |
| 二进制文件 | 图片（`.png`、`.jpg`、`.gif`、`.svg`、`.ico`、`.webp`）、字体（`.ttf`、`.woff`、`.woff2`、`.eot`）、编译后的二进制文件（`.exe`、`.dll`、`.so`、`.dylib`、`.o`、`.a`、`.class`、`.pyc`、`.pyo`、`.wasm`） |
| 自引用 | `docs/development-status/summaries/` 本身、`.skill-package/` |

项目可通过在 `docs/development-status/00-index.md` 中记录覆盖规则来自定义这些默认值。

## 模板：代码文件

适用于源代码文件（`.py`、`.ts`、`.tsx`、`.js`、`.jsx`、`.go`、`.rs`、`.java`、`.kt`、`.swift`、`.c`、`.cpp`、`.h`、`.rb`、`.php`、`.scala`、`.cs`、`.sh`、`.bash`、`.zsh`、`.sql` 等）：

```markdown
# `<从项目根目录的相对路径>`

## 功能
<1-3 句话描述该文件做什么。聚焦其核心职责——做什么，而非怎么做。>

## 关系
### 调用 / 引用
- `<相对路径到项目文件>` — <简要描述使用了什么>
- `<模块.类.方法()>` — <简要描述外部 API 调用>
- `<外部包>` — <提供的内容>

### 提供
- `<类名>` — <简要描述类的职责>
- `<函数名>()` — <简要描述函数>
- `<常量名>` — <简要描述>

### 被依赖
- `<相对路径到项目文件>`
- `<相对路径到另一个消费者>`
```

### 字段规则

| 字段 | 何时包含 | 何时省略 |
|------|---------|---------|
| `## 功能` | 始终 | 从不 |
| `### 调用 / 引用` | 文件导入、引用或依赖其他项目文件或外部库 | 文件零依赖（罕见） |
| `### 提供` | 文件定义了供外部消费的公共类、函数或常量 | 文件是无公共 API 的叶子脚本（例如仅运行的一次性入口点） |
| `### 被依赖` | 已知有其他项目文件依赖此文件 | 无已知依赖者（全新文件常见） |
| `## 关键词` | 代码文件始终 | 非代码文件从不 |

- **`## 关系` 下的三个子部分都是可选的。** 如果文件没有对外依赖，完全省略 `### 调用 / 引用`——不要写"无"或"暂无"。
- **关系部分中的文件路径不使用 `./` 前缀。** 写 `src/utils/helper.py`，而非 `./src/utils/helper.py`。
- **标题中的文件路径使用反引号包裹的相对路径** — `` `src/utils/helper.py` `` — 使其在 Markdown 中以内联代码形式呈现。

### 关键词部分

```markdown
## 关键词
### 类
- `类名`
- `另一个类`

### 函数
- `函数名`
- `另一个函数`

### 常量
- `常量名`
```

- **仅**列出名称——无描述、无参数列表、无行号。
- 按类别分组（类、函数、常量、接口、类型别名等）。根据语言需要添加类别。
- 如某类别无条目，省略该子部分。
- 每个类别内的排序：字母顺序或定义顺序均可，只要文件内保持一致。

## 模板：非代码文件

适用于配置、构建和基础设施文件（`Dockerfile`、`Makefile`、`package.json`、`docker-compose.yml`、`tsconfig.json`、`*.toml`、`*.cfg`、`*.ini`、`*.yaml`、`*.yml`、`CMakeLists.txt`、SQL 迁移脚本、GitHub Actions 工作流等）：

```markdown
# `<从项目根目录的相对路径>`

## 功能
<此配置/构建/基础设施文件做什么。>

## 关系
### 引用
- `<路径>` — <简要描述>

### 被依赖
- `<路径>`
```

### 与代码模板的区别

- **没有 `## 关键词` 部分。** 非代码文件不定义类或函数。
- **更简单的关系模型。** 使用 `### 引用` 代替 `### 调用 / 引用`，`### 被依赖` 保持不变。
- **所有块仍然是可选的。** 省略不适用的部分。

---

## 模板：文档和静态数据文件

适用于 Markdown 文档、JSON 数据文件、CSV 文件和其他静态内容：

```markdown
# `<从项目根目录的相对路径>`

## 功能
<此文件包含的内容及其用途。>
```

- **仅 `## 功能` 是必需的。** 仅当文件有有意义的结构性依赖时才添加 `## 关系` 块。
- **没有 `## 关键词` 部分。**

---

## 编写指南

1. **保持简洁。** 每个描述应为 1-3 句话。摘要的目的是帮助 agent 定位正确的文件——而非替代阅读文件本身。
2. **描述做什么，而非怎么做。** "处理用户认证和会话管理"是好的描述。"使用 bcrypt 12 轮哈希并将 session 以 24h TTL 存储在 Redis 中"属于架构文档或代码注释。
3. **保持关系真实。** 仅记录你在代码或其他摘要中实际观察到的依赖关系。不要猜测。
4. **积极更新。** 每次代码变更后更新摘要。过时的摘要比没有摘要更糟——它会产生主动误导。
5. **探索时交叉验证。** 阅读摘要时，如果某些内容与你从其他地方看到的似乎不一致，对照源码验证并修复摘要。

## 关系追踪协议

摘要中的关系是**双向的**。当文件 `A` 依赖于文件 `B` 时：

- `A.md` 在 `### 调用 / 引用` 下记录 `B`
- `B.md` 在 `### 被依赖` 下记录 `A`

### 每次代码修改时

1. **更新每个被修改文件的摘要。** 这是强制性的——即使对于单行修复也是如此。
2. **检查被修改文件中的依赖变更：**
   - **新增对文件 X 的导入/引用** → 将当前文件添加到 `X.md` 的 `### 被依赖` 部分。
   - **移除对文件 Y 的导入/引用** → 从 `Y.md` 的 `### 被依赖` 部分中移除当前文件。
3. **文件删除时：** 移除摘要文件，并从所有其他摘要中移除对被删除文件的引用。
4. **文件重命名/移动时：** 将摘要文件移动到新路径，并更新所有相关摘要中的路径引用。

### 收敛，而非完美

双向一致性以**增量方式**维护。当文件 `A` 被修改时，只有 `A.md` 和 `A` 的直接依赖的摘要被更新。依赖 `A` 的文件（下游消费者）的摘要在这些文件自身下次被修改时才会更新。这意味着对于被频繁使用的文件，`### 被依赖` 部分可能不完整——这是可以接受的，且随着开发的继续会自我修正。
